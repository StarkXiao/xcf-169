# 守钟人日记与经典校时联动模块分析

## 1. 模块概述

本模块由 **守钟人日记系统 (KeeperDiarySystem)** 与 **经典校时模式** 联动组成，通过游戏进度解锁日记章节，章节提供的校准效果反向影响游戏玩法，形成"游戏→解锁→增益→再游戏"的循环。

### 核心文件

| 系统 | 文件路径 | 主要职责 |
|------|----------|----------|
| 守钟人日记系统 | `src/game/KeeperDiarySystem.ts` | 日记条目管理、解锁判定、效果叠加、统计记录 |
| 钟声评价系统 | `src/game/BellChimeEvaluationSystem.ts` | 校时质量评估、等级判定(S/A/B/C/D/F)、回放记录 |
| 钟声谱面系统 | `src/game/BellChimeSystem.ts` | 钟声预设管理、音符生成、触发控制 |
| 主游戏逻辑 | `src/game/Game.tsx` | 经典校时模式核心玩法、状态流转 |
| 应用入口 | `src/App.tsx` | 全局协调、结算分发、系统间数据流转 |
| 结算面板 | `src/ui/GameOverPanel.tsx` | 结果展示、章节文案、新解锁提示 |
| 工坊系统 | `src/game/WorkshopSystem.ts` | 累计分数、装备解锁、材料效果 |
| 钟楼名册 | `src/game/ClocktowerRecordSystem.ts` | 游戏记录、最佳成绩、统计汇总 |

### 3.2 解锁条件评估机制

**位置**: `KeeperDiarySystem.ts:532-557`

```typescript
private evaluateCondition(condition: DiaryUnlockCondition): boolean {
  const { type, value } = condition
  const stats = this.state.stats

  switch (type) {
    case 'score_reached':
      return workshopSystem.getTotalScoreEarned() >= value  // ⚠️ 用的是工坊分数
    case 'perfect_clear':
      return stats.perfectClears >= value
    case 'consecutive_wins':
      return stats.bestConsecutiveWins >= value            // ⚠️ 用的是历史最佳
    case 'speed_clear':
      return stats.fastestClearSeconds <= value
    case 'no_fault_clear':
      return stats.noFaultClears >= value
    case 'total_plays':
      return stats.totalPlays >= value
    // ... 其他类型暂未实现
  }
}
```

**多条件逻辑**: 每个日记条目可配置多个解锁条件，通过 `unlockAllRequired` 控制：
- `true` (默认): **所有条件都满足** 才解锁
- `false`: **满足任一条件** 即可解锁

## 4. 结算切换逻辑

### 4.1 经典模式结算流程

**位置**: `Game.tsx:172-291`

```
游戏结束 (成功/失败)
    │
    ├─ 停止计时器
    ├─ 播放音效 (level_success / level_fail)
    ├─ 播放游戏结束音效
    ├─ 计算风暴分数影响
    │
    ├─ [成功] 播放胜利动画 + 钟声
    ├─ [失败] 播放失败动画
    │
    ├─ 计算基础分数 (calculateClassicScore)
    │   ├─ accuracyBonus = max(0, (360 - diffMinutes) * 2)
    │   ├─ timeBonus = floor(remaining * 5 * timeBonusMultiplier)
    │   ├─ perfectBonus = diffMinutes === 0 ? 1000 : 0
    │   └─  × diaryScoreMultiplier × difficultyMultiplier
    │
    ├─ 风暴影响修正 (+ stormImpact)
    │
    ├─ 钟声评价 (BellChimeEvaluationSystem.evaluate)
    │   ├─ accuracyScore (0-350)
    │   ├─ speedScore (0-350)
    │   ├─ flowScore (0-350)
    │   ├─ 总分 → 等级判定 (S/A/B/C/D/F)
    │   └─ 计算评价加成 (accuracyBonus + speedBonus + flowBonus + gradeBonus)
    │
    ├─ 最终分数 = 基础分 + 评价加成
    │
    ├─ 工坊记录分数
    ├─ 保存回放
    │
    └─ [成功且非巡逻模式]
        ├─ 延迟 2.5s
        └─ 进入观光模式 → 观光结束后触发 onGameEnd
```

### 4.2 App层结算分发

**位置**: `App.tsx:75-128`

```typescript
const handleGameEnd = useCallback((result: AnyGameResult) => {
  setGameResult(result)
  
  // 1. 工坊记录
  workshopSystem.recordGameScore(result.score)
  
  // 2. 钟声系统记录
  bellChimeSystem.recordScore(result.score)
  bellChimeSystem.resetTriggers()
  
  // 3. 日记系统记录 (关键！)
  const timeUsed = totalTime - result.timeLeft
  const isPerfect = result.score >= 2500        // ⚠️ 硬编码阈值
  const hadFaults = result.score < 2000 && result.success  // ⚠️ 分数反推
  keeperDiarySystem.recordGameResult(
    result.success,
    result.score,
    Math.max(1, timeUsed),
    isPerfect,
    hadFaults
  )
  
  // 4. 钟楼名册记录
  clocktowerRecordSystem.recordGameResult(...)
  
  // 5. 成就系统记录
  achievementSystem.recordGameResult(...)
  
  goTo('result')
}, [])
```

## 5. 章节判定风险点分析

### 🔴 风险等级说明
- 🔴 **高风险**: 直接影响核心玩法体验或经济平衡
- 🟡 **中风险**: 影响玩家感知或进度流畅度
- 🟢 **低风险**: 潜在问题但影响有限

---

### 🔴 风险1: 双分数系统不一致

**问题描述**:
系统中存在**两个独立的累计分数**，用途不同：

| 分数系统 | 存储位置 | 计算方式 | 用途 |
|----------|----------|----------|------|
| 工坊分数 | `workshopSystem.totalScoreEarned` | 原始最终得分 (含评价加成) | 解锁装备/材料、日记 score_reached 条件 |
| 日记分数 | `keeperDiarySystem.totalDiaryScore` | 原始得分 × diary效果乘数 | 仅日记系统内部统计 |

**触发场景**:
- 玩家激活了日记的得分加成效果 (如 diary_001 +5%, diary_006 +15%)
- 此时 `totalDiaryScore` 增长比 `totalScoreEarned` 快
- 但日记解锁条件 `score_reached` 用的是 **工坊分数**，不是日记自己的分数

**潜在影响**:
1. 玩家困惑："我的日记分数已经够了，为什么还没解锁？"
2. 设计意图不明确：日记分数的存在意义是什么？
3. 平衡性问题：如果未来用日记分数做解锁，可能导致进度过快

**相关代码**:
- `KeeperDiarySystem.ts:538`: `workshopSystem.getTotalScoreEarned()`
- `KeeperDiarySystem.ts:614-615`: `finalScore = Math.floor(score * effects.scoreMultiplier)`

---

### 🔴 风险2: "完美校准"判定标准不统一

**问题描述**:
系统中至少有 **3种"完美"的判定标准**，彼此不一致：

| 判定位置 | 判定标准 | 用途 |
|----------|----------|------|
| App.tsx:90 | `result.score >= 2500` | 传入日记系统的 isPerfect 参数 |
| GameOverPanel.tsx:36 | `result.score >= 2500` | 结算面板的 perfect 文案选择 |
| ClocktowerRecordSystem.ts:121 | `result.score >= 2500` | 记录的 isPerfect 标记 |
| BellChimeEvaluationSystem | `totalScore >= 900` 且 success | S级评价 (评价体系内的"完美") |
| GearSystem | `diffMinutes === 0` | 零偏差的"时间完美对齐" |

**核心问题**:
1. **硬编码阈值**: `2500` 分散在多个文件中，没有统一常量
2. **含义混淆**: 分数达标 vs 零偏差 vs S级评价，都叫"完美"
3. **日记解锁条件歧义**: `perfect_clear` 到底是哪个"完美"？

**当前实现** (`KeeperDiarySystem.ts:598-599`):
```typescript
if (isPerfect) {
  this.state.stats.perfectClears += 1
}
```

这里的 `isPerfect` 是从 App 层传入的，即 **分数 ≥ 2500**，而不是零偏差。

**潜在影响**:
- 玩家以为"零偏差"就是完美校准，但实际要看分数
- 难度变化会影响分数，间接影响"完美"判定
- 多处硬编码，修改时容易遗漏

---

### 🔴 风险3: hadFaults 通过分数反推，不准确

**问题描述**:
`hadFaults` 参数用于统计 `noFaultClears`（无故障通关次数），但它的计算方式是：

```typescript
// App.tsx:90
const hadFaults = result.score < 2000 && result.success
```

**用分数反推是否有故障**，而不是使用实际的故障计数。

**实际故障数据**: `bellEvaluation.details.faultCount` 才是真实的故障次数。

**潜在影响**:
1. **误判**: 分数低不一定是因为故障，可能是操作慢、偏差大
2. **漏判**: 有故障但分数仍然很高的情况（例如快速修复了故障）
3. **不可靠**: 难度、装备、日记效果都会影响分数，导致阈值失真
4. **影响 diary_004 解锁**: "风雨无阻"需要无故障通关3次

**相关代码**:
- `App.tsx:90-97`: hadFaults 计算
- `KeeperDiarySystem.ts:602-604`: noFaultClears 统计

---

### 🟡 风险4: "当前章节"定义歧义

**问题描述**:
系统中有两个"当前"概念：

| 概念 | 获取方式 | 含义 |
|------|----------|------|
| currentEntryId | `this.state.currentEntryId` | 用户最后**阅读**的日记 |
| currentDiaryChapter | `getCurrentDiaryChapter()` | 最后**解锁**的日记（最新章节） |

**结算时使用的是后者**（最后解锁的），但用户可能以为是自己正在读的那章。

**相关代码**:
- `GameOverPanel.tsx:36-37`:
  ```typescript
  const currentChapter = keeperDiarySystem.getCurrentDiaryChapter()
  const settlementText = keeperDiarySystem.getSettlementText(currentChapter?.id || null)
  ```
- `KeeperDiarySystem.ts:621-627`:
  ```typescript
  getSettlementText(entryId?: DiaryEntryId | null): DiarySettlementText {
    const id = entryId || this.state.currentEntryId  // 有参数用参数，没参数用当前阅读
    // ...
  }
  ```

**潜在影响**:
1. 玩家阅读了第3章，但解锁了第5章，结算显示第5章文案
2. 玩家可能困惑："我明明在看第3章，怎么结算用的第5章的文案？"
3. 章节叙事的沉浸感被打破

### 🟡 风险5: 连胜解锁使用"历史最佳"而非"当前连胜"

**问题描述**:
`diary_006` 的解锁条件是 `consecutive_wins >= 5`，但实际检查的是 **`bestConsecutiveWins`**（历史最佳连胜记录）。

```typescript
// KeeperDiarySystem.ts:542-543
case 'consecutive_wins':
  return stats.bestConsecutiveWins >= value
```

**连胜统计逻辑** (`KeeperDiarySystem.ts:589-611`):
- 成功: `consecutiveWins += 1`，并更新 `bestConsecutiveWins`
- 失败: `consecutiveWins = 0`（重置当前连胜）

**设计意图分析**:
可能是有意为之——让玩家曾经达到过5连胜就算解锁。但条件名 `consecutive_wins` 有歧义。

**潜在影响**:
1. 玩家理解偏差：以为需要"当前连胜5次"，实际是"历史最佳5连胜"
2. 解锁难度降低：达成一次就永久解锁，不需要保持

### 🟡 风险7: 分数乘数的多次叠加

**问题描述**:
最终得分在计算过程中经过**多次乘法**，透明度不足：

```
基础分 (accuracy + time + perfect)
  × 日记效果乘数 (scoreMultiplier)  → Game.calculateClassicScore
  × 难度乘数 (difficultyMultiplier)  → Game.calculateClassicScore
  + 评价加成 (totalBonus)             → Game.handleGameEnd
  + 风暴修正 (stormImpact)            → Game.handleGameEnd
  = 最终得分 (finalScore)
```

然后：
- `workshopSystem.recordGameScore(finalScore)` → 工坊累计
- `keeperDiarySystem.recordGameResult(..., score, ...)` → 日记又乘一次？

**检查日记系统**:
```typescript
// KeeperDiarySystem.ts:613-615
const effects = this.getEffects()
const finalScore = Math.floor(score * effects.scoreMultiplier)
this.state.totalDiaryScore += finalScore
```

⚠️ **是的，日记系统内部又乘了一次 scoreMultiplier！**

但传入的 `score` 本身已经在 Game 层乘过一次了。

**潜在影响**:
1. **双重乘算**: 日记加成效果被应用了两次（Game层一次，KeeperDiarySystem层一次）
2. 日记分数膨胀速度远超预期
3. 虽然日记分数目前只作内部统计，但如果未来用于解锁，会出问题

### 🟢 风险8: 结算时系统调用顺序依赖

**问题描述**:
`App.handleGameEnd()` 中各系统的调用顺序是：

```
1. workshopSystem.recordGameScore()   // 先更新工坊
2. bellChimeSystem.recordScore()
3. keeperDiarySystem.recordGameResult()  // 后更新日记
   └─ checkUnlocks()  // 内部会检查 workshopSystem 的分数
```

由于日记的 `score_reached` 条件依赖工坊分数，所以**工坊必须先更新**。

当前顺序是对的，但这是一个**隐式依赖**，没有注释说明。

**潜在影响**:
1. 未来重构时如果调换顺序，会导致解锁延迟一帧
2. 新人维护时容易踩坑

---

### 🟢 风险9: 多模式下的统计口径不一致

**问题描述**:
`keeperDiarySystem.recordGameResult()` 对所有游戏模式都生效，但：

| 模式 | 分数计算方式 | 时间统计 | 故障判定 |
|------|-------------|----------|----------|
| 经典模式 | 标准公式 | timeLeft | 不准确 (分数反推) |
| 巡逻模式 | 累计多时段 | 单时段剩余时间 | 未传入 |
| 多钟塔模式 | 独立计算 | 未确认 | 未确认 |
| 肉鸽模式 | 完全不同 | 不相关 | 不相关 |

**巡逻模式的问题**:
- 巡逻模式的 `result.score` 是多时段累计分
- `timeLeft` 是最后一个时段的剩余时间
- `hadFaults` 还是用老的分数反推逻辑
- 但巡逻模式的故障机制和经典模式不同

**潜在影响**:
1. 巡逻模式的游戏统计数据不准确
2. 玩家可以通过巡逻模式"刷"日记解锁
3. 统计数据的可比性降低

## 7. 总结与建议

### 高优先级修复建议

| 风险 | 建议 | 预估影响 |
|------|------|----------|
| 🔴 双分数系统不一致 | 明确两个分数的定位，统一解锁条件使用的分数来源 | 核心平衡 |
| 🔴 完美判定不统一 | 提取 `PERFECT_SCORE_THRESHOLD` 常量，统一"完美"的命名和含义 | 代码质量 + 玩家体验 |
| 🔴 hadFaults反推不准确 | 使用 `bellEvaluation.details.faultCount` 直接判断 | 统计准确性 |
| 🟡 奖励类型不完整 | 实现 badge/material/tool/bell_preset 奖励的发放逻辑 | 核心玩法完整性 |
| 🟡 双重乘算问题 | 确认日记分数是否需要独立乘算，避免重复应用加成 | 经济平衡 |

### 中优先级优化建议

1. **统一"当前章节"概念**：明确结算时使用哪个，或者增加玩家选择
2. **命名澄清**：`consecutive_wins` 改为 `best_consecutive_wins` 或调整判定逻辑
3. **增加注释**：说明各系统调用顺序的依赖关系
4. **巡逻模式适配**：确保巡逻模式的统计数据正确录入日记系统

### 低优先级改进建议

1. 增加解锁条件的可视化进度
2. 章节结算文案与当前阅读的日记关联
3. 增加日记章节的叙事连贯性设计
